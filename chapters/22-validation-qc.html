33  Validation and Quality Control

33.1 22.1 Introduction to Validation in Clinical Programming

The stakes for data integrity in clinical trials are extraordinarily high. A single computational error in efficacy analysis could delay a drug approval by years; a missed safety signal could harm patients. Regulatory agencies like the FDA, EMA, and ICH recognize this, embedding validation requirements into guidance documents like 21 CFR Part 11 (“Electronic Records; Electronic Signatures”), ICH GCP (“Good Clinical Practice”), and GAMP 5 (“Guidance for the Use of Prototyping in the Pharmaceutical Industry”).

Validation in clinical programming means ensuring that: 1. Software and analytical code produce correct, reproducible results. 2. Data transformations are accurate and traceable. 3. Statistical analyses comply with the statistical analysis plan (SAP). 4. Regulatory submissions contain complete, auditable records of all computational steps.

This chapter presents practical workflows and R tools to validate ADaM datasets, statistical analyses, and clinical outputs. We focus on three core strategies: double programming, unit testing, and automated dataset comparison.

33.2 22.2 Regulatory and Quality Frameworks

33.2.1 22.2.1 21 CFR Part 11: Electronic Records

21 CFR Part 11 is the FDA rule governing electronic records and signatures in regulated industries. Key requirements include:

  • Audit Trail: Permanent, computer-generated records of all data entries, deletions, and modifications.
  • Data Integrity: Systems must ensure that data remain accurate and complete.
  • User Authentication: Only authorized users can access and modify data.
  • Validation of Systems: Software used to process clinical data must be validated.

For R programmers, this means: - Version control (git) provides audit trails (who, when, what). - Reproducible scripts eliminate manual, untracked manipulations. - Unit tests provide evidence of correctness. - Change logs document modifications and rationale.

33.2.2 22.2.2 ICH GCP

ICH GCP (Good Clinical Practice) requires that: - All data be traceable to source documentation. - Statistical analyses are performed according to the SAP, which is finalized before database lock. - Any deviations from the SAP are documented and justified.

33.2.3 22.2.3 GAMP 5: Software Validation Phases

GAMP 5 defines a lifecycle for software validation:

Phase Activity Evidence
IQ (Installation Qualification) Verify software is installed correctly and meets specifications Installation logs, vendor documentation
OQ (Operational Qualification) Verify software operates as specified under normal conditions Test scripts, results
PQ (Performance Qualification) Verify software meets user requirements in the actual operational environment Test cases with real/realistic data

For R package validation, this translates to: - IQ: Confirm R version, package versions, and dependencies via renv.lock. - OQ: Run unit tests to verify functions work correctly. - PQ: Test with realistic clinical data (e.g., pharmaversesdtm) to verify end-to-end workflows.

33.3 22.3 Double Programming and Reconciliation

Double programming is the gold standard in clinical data validation: two independent programmers perform the same analysis using independent code, and results are compared cell-by-cell.

33.3.1 22.3.1 Double Programming Workflow

Raw Data (SDTM)
    ↓
Programmer A writes ADaM code → ADaM_A
Programmer B writes ADaM code → ADaM_B
    ↓
Compare ADaM_A vs ADaM_B using diffdf
    ↓
Identify discrepancies → Root cause analysis
    ↓
Reconciliation meeting → Agree on correct approach
    ↓
Final ADaM dataset

33.3.2 22.3.2 Example: Double Programming for Change from Baseline

Programmer A’s approach (using dplyr):

# Programmer A: Calculate change from baseline
library(dplyr)

adlb_a <- lab_data %>%
  arrange(SUBJID, LBTESTCD, VISITNUM) %>%
  group_by(SUBJID, LBTESTCD) %>%
  mutate(
    AVAL = LBSTRESN,
    ABASE = first(LBSTRESN),  # First value = baseline
    CHG = AVAL - ABASE,
    PCHG = (CHG / ABASE) * 100
  ) %>%
  ungroup()

Programmer B’s approach (using data.table):

# Programmer B: Calculate change from baseline
library(data.table)

lab_dt <- as.data.table(lab_data)

adlb_b <- lab_dt[
  order(SUBJID, LBTESTCD, VISITNUM),
  c("AVAL", "ABASE", "CHG", "PCHG") := {
    aval <- LBSTRESN
    abase <- LBSTRESN[1]  # First value = baseline
    chg <- aval - abase
    pchg <- (chg / abase) * 100
    list(AVAL = aval, ABASE = abase, CHG = chg, PCHG = pchg)
  },
  by = .(SUBJID, LBTESTCD)
]

33.3.3 22.3.3 Comparing Outputs with diffdf

The diffdf package (from admiral) compares two datasets and flags all differences:

if (requireNamespace("diffdf", quietly = TRUE)) {
  library(diffdf)

  # Compare Programmer A's and B's outputs
  comparison <- diffdf(
    base = adlb_a,
    compare = adlb_b,
    keys = c("SUBJID", "LBTESTCD", "VISITNUM")
  )

  # View summary of differences
  summary(comparison)

  # Extract detailed differences
  comparison$comparison_result
}
#> NULL

Output might show: - Structure differences: Extra/missing columns - Value differences: Cell-level discrepancies - Key differences: Records in one dataset but not the other

If differences exist, the team meets to identify the root cause: - Rounding rules different? - Baseline definition different (first visit vs. screening)? - Missing data handling different?

Once reconciled, the agreed-upon approach is coded and validated.

33.4 22.4 Unit Testing with testthat

Unit testing is a software development practice in which small, isolated code units are tested in isolation. In clinical programming, unit tests verify that: - Data transformations work correctly. - Edge cases (missing data, extreme values) are handled properly. - Statistical calculations match known results.

33.4.1 22.4.1 testthat Package Basics

The testthat package provides a framework for writing tests:

33.4.3 22.4.3 Comprehensive Test Suite

Create a file tests/test-adae.R:

if (requireNamespace("testthat", quietly = TRUE)) {
  library(testthat)
  library(dplyr)

  # Test: Serious AE flag assignment
  test_that("AESRDFN flag assigned correctly", {
    test_ae <- data.frame(
      AESER = c("Y", "N", "Y", NA),
      expected_aesrdfn = c("Serious", "Non-serious", "Serious", NA)
    )

    # Function to test
    assign_aesrdfn <- function(aeser) {
      ifelse(aeser == "Y", "Serious",
             ifelse(aeser == "N", "Non-serious", NA))
    }

    result <- assign_aesrdfn(test_ae$AESER)
    expect_equal(result, test_ae$expected_aesrdfn)
  })

  # Test: AE causality assignment
  test_that("AEACNOTH populated when AEACN is 'Other'", {
    test_ae <- data.frame(
      AEACN = c("DOSE INCREASED", "OTHER"),
      AEACNOTH = c(NA, "Custom action")
    )

    adae <- test_ae %>%
      mutate(
        check_aeacnoth = ifelse(AEACN == "OTHER" & is.na(AEACNOTH), 1, 0)
      )

    expect_true(all(adae$check_aeacnoth == 0))
  })

  # Test: Missing data handling
  test_that("Missing values handled correctly in analysis", {
    test_data <- data.frame(
      SUBJID = c(1, 2, 3),
      AEVAL = c(100, NA, 110)
    )

    # Filter out missing analyses
    analyzed <- test_data %>%
      filter(!is.na(AEVAL))

    expect_equal(nrow(analyzed), 2)
    expect_false(any(is.na(analyzed$AEVAL)))
  })
}

33.5 22.5 Validation Documentation

33.5.1 22.5.1 Validation Plan

A Validation Plan is a document that outlines the scope, approach, and acceptance criteria for validating a software system. For an R-based ADaM generation system, it might include:

Section 1: System Overview - System name: “ADaM Dataset Generation System (R-based)” - Scope: SDTM → ADaM conversion for efficacy and safety variables - Users: Biostatisticians, statistical programmers

Section 2: Validation Approach - IQ: Verify R environment, package versions, dependencies - OQ: Unit tests for core functions (change from baseline, AESI assignment, etc.) - PQ: End-to-end tests with realistic clinical datasets (pharmaversesdtm)

Section 3: Test Cases - Change from baseline: 15 test cases (normal values, missing data, outliers) - AE flag assignment: 8 test cases (serious, related, action taken) - Baseline definitions: 5 test cases (different visit selections)

Section 4: Acceptance Criteria - All test cases must pass without errors or warnings - Code coverage ≥90% - Double programming reconciliation completed (0 discrepancies)

Section 5: Evidence - Test results (logs) - Code coverage reports - Signed reconciliation forms

33.7 22.7 R Package Validation: pharmaR Framework

For organizations using custom R packages in clinical programming, pharmaR provides a framework for validating packages according to GAMP 5 and ICH standards.

The pharmaR package (from Roche/Pharmaverse) assesses package maturity:

  print(pkg_assessment)

Output includes: - Download Statistics: CRAN downloads per month (higher = more tested) - Maintenance: Time since last update - Testing: Unit test coverage, CI/CD (GitHub Actions, etc.) - Documentation: Vignettes, man pages

33.7.2 22.7.2 PHUSE Guidelines

The PHUSE (Pharmaceutical Users Software Exchange) organization publishes guidelines for validating R packages. Key recommendations:

  1. Choose Well-Maintained Packages: Prefer packages with active development and large user bases.
  2. Document Dependencies: Use renv to lock package versions.
  3. Test with Clinical Data: Run packages against realistic ADaM datasets.
  4. Maintain Change Logs: Track package version updates and testing results.

33.8 22.8 Controlled R Environments with renv

33.8.1 22.8.1 Why renv Matters for Validation

Without renv, reproducibility is fragile: - A colleague’s system has a different R version. - A new package update changes behavior. - Six months later, you can’t re-run an analysis.

renv solves this by “freezing” your project’s R environment:

Project A (2024-01-15):
  R 4.3.2
  dplyr 1.1.0
  ggplot2 3.4.1

Project B (2023-06-20):
  R 4.2.0
  dplyr 1.0.10
  ggplot2 3.3.6

When you open Project A, renv automatically loads the frozen environment, ensuring reproducibility.

33.8.2 22.8.2 renv Workflow

Initialize renv in a project:

library(renv)
renv::init()

This creates: - renv.lock: JSON file listing all packages and versions - renv/: Isolated library directory - .Rprofile: Startup script activating renv

Install/update packages:

# Install a package (recorded in renv.lock)
install.packages("dplyr")

# Update renv.lock with new snapshot
renv::snapshot()

Restore environment on another machine:

# On colleague's machine or after git clone
renv::restore()  # Installs all packages from renv.lock

33.9 22.9 CDISC Compliance Checks

33.9.1 22.9.1 Pinnacle 21 (Conceptual Overview)

Pinnacle 21 Community is a web-based tool (by FDA/CDISC) that validates SDTM and ADaM datasets against CDISC standards. While not an R package, understanding its checks helps us validate in R.

Key Pinnacle 21 checks: - SDTM: Variable names (STUDYID, SITEID, SUBJID), domains (DM, AE, LB), controlled terminology - ADaM: Proper flags (ITT, EFFFL, SAFFL), analysis variables (AVAL, BASE, CHG)

33.9.2 22.9.2 Basic CDISC Compliance Checks in R

Create a function to validate ADaM variables:

# Validate CDISC ADaM structure
validate_adam <- function(df) {
  errors <- list()

  # Check required variables
  required_vars <- c("STUDYID", "USUBJID", "AVAL", "AVISIT")
  missing_vars <- setdiff(required_vars, names(df))
  if (length(missing_vars) > 0) {
    errors$missing_vars <- missing_vars
  }

  # Check no missing AVAL for analysis population
  if (any(is.na(df$AVAL))) {
    errors$missing_aval <- "AVAL contains missing values"
  }

  # Check AVISIT is controlled terminology
  valid_visits <- c("Baseline", "Week 2", "Week 4", "Week 8", "Week 12")
  invalid_visits <- setdiff(unique(df$AVISIT), valid_visits)
  if (length(invalid_visits) > 0) {
    errors$invalid_visits <- invalid_visits
  }

  # Return results
  if (length(errors) == 0) {
    cat("✓ ADaM dataset is CDISC-compliant\n")
    return(invisible(TRUE))
  } else {
    cat("✗ ADaM compliance errors found:\n")
    print(errors)
    return(invisible(FALSE))
  }
}

# Use the function (adlb_a created in section 22.3.2)

33.10 22.10 Audit Trails and Traceability Logging

33.10.1 22.10.1 Why Audit Trails Matter

21 CFR Part 11 requires a permanent record: who accessed/modified data, what changed, when it changed. While git handles code changes, logging handles data processing.

# Validation and Quality Control


## 22.1 Introduction to Validation in Clinical Programming

The stakes for data integrity in clinical trials are extraordinarily high. A single computational error in efficacy analysis could delay a drug approval by years; a missed safety signal could harm patients. Regulatory agencies like the FDA, EMA, and ICH recognize this, embedding validation requirements into guidance documents like **21 CFR Part 11** ("Electronic Records; Electronic Signatures"), **ICH GCP** ("Good Clinical Practice"), and **GAMP 5** ("Guidance for the Use of Prototyping in the Pharmaceutical Industry").

**Validation** in clinical programming means ensuring that:
1. Software and analytical code produce correct, reproducible results.
2. Data transformations are accurate and traceable.
3. Statistical analyses comply with the statistical analysis plan (SAP).
4. Regulatory submissions contain complete, auditable records of all computational steps.

This chapter presents practical workflows and R tools to validate ADaM datasets, statistical analyses, and clinical outputs. We focus on three core strategies: **double programming**, **unit testing**, and **automated dataset comparison**.

## 22.2 Regulatory and Quality Frameworks

### 22.2.1 21 CFR Part 11: Electronic Records

**21 CFR Part 11** is the FDA rule governing electronic records and signatures in regulated industries. Key requirements include:

- **Audit Trail**: Permanent, computer-generated records of all data entries, deletions, and modifications.
- **Data Integrity**: Systems must ensure that data remain accurate and complete.
- **User Authentication**: Only authorized users can access and modify data.
- **Validation of Systems**: Software used to process clinical data must be validated.

For R programmers, this means:
- Version control (git) provides audit trails (who, when, what).
- Reproducible scripts eliminate manual, untracked manipulations.
- Unit tests provide evidence of correctness.
- Change logs document modifications and rationale.

### 22.2.2 ICH GCP

**ICH GCP** (Good Clinical Practice) requires that:
- All data be traceable to source documentation.
- Statistical analyses are performed according to the SAP, which is finalized before database lock.
- Any deviations from the SAP are documented and justified.

### 22.2.3 GAMP 5: Software Validation Phases

**GAMP 5** defines a lifecycle for software validation:

| Phase | Activity | Evidence |
|-------|----------|----------|
| **IQ (Installation Qualification)** | Verify software is installed correctly and meets specifications | Installation logs, vendor documentation |
| **OQ (Operational Qualification)** | Verify software operates as specified under normal conditions | Test scripts, results |
| **PQ (Performance Qualification)** | Verify software meets user requirements in the actual operational environment | Test cases with real/realistic data |

For R package validation, this translates to:
- **IQ**: Confirm R version, package versions, and dependencies via renv.lock.
- **OQ**: Run unit tests to verify functions work correctly.
- **PQ**: Test with realistic clinical data (e.g., pharmaversesdtm) to verify end-to-end workflows.

## 22.3 Double Programming and Reconciliation

**Double programming** is the gold standard in clinical data validation: two independent programmers perform the same analysis using independent code, and results are compared cell-by-cell.

### 22.3.1 Double Programming Workflow

```
Raw Data (SDTM)
    ↓
Programmer A writes ADaM code → ADaM_A
Programmer B writes ADaM code → ADaM_B
    ↓
Compare ADaM_A vs ADaM_B using diffdf
    ↓
Identify discrepancies → Root cause analysis
    ↓
Reconciliation meeting → Agree on correct approach
    ↓
Final ADaM dataset
```

### 22.3.2 Example: Double Programming for Change from Baseline

```{r}
#| include: false
library(dplyr)
library(pharmaversesdtm)
data(lb)
lab_data <- lb %>% rename(SUBJID = USUBJID)
```

**Programmer A's approach** (using dplyr):

```{r}
#| echo: true

# Programmer A: Calculate change from baseline
library(dplyr)

adlb_a <- lab_data %>%
  arrange(SUBJID, LBTESTCD, VISITNUM) %>%
  group_by(SUBJID, LBTESTCD) %>%
  mutate(
    AVAL = LBSTRESN,
    ABASE = first(LBSTRESN),  # First value = baseline
    CHG = AVAL - ABASE,
    PCHG = (CHG / ABASE) * 100
  ) %>%
  ungroup()
```

**Programmer B's approach** (using data.table):

```{r}
#| echo: true

# Programmer B: Calculate change from baseline
library(data.table)

lab_dt <- as.data.table(lab_data)

adlb_b <- lab_dt[
  order(SUBJID, LBTESTCD, VISITNUM),
  c("AVAL", "ABASE", "CHG", "PCHG") := {
    aval <- LBSTRESN
    abase <- LBSTRESN[1]  # First value = baseline
    chg <- aval - abase
    pchg <- (chg / abase) * 100
    list(AVAL = aval, ABASE = abase, CHG = chg, PCHG = pchg)
  },
  by = .(SUBJID, LBTESTCD)
]
```

### 22.3.3 Comparing Outputs with diffdf

The **diffdf** package (from admiral) compares two datasets and flags all differences:

```{r}
#| echo: true

if (requireNamespace("diffdf", quietly = TRUE)) {
  library(diffdf)

  # Compare Programmer A's and B's outputs
  comparison <- diffdf(
    base = adlb_a,
    compare = adlb_b,
    keys = c("SUBJID", "LBTESTCD", "VISITNUM")
  )

```

Output might show:
- **Structure differences**: Extra/missing columns
- **Value differences**: Cell-level discrepancies
- **Key differences**: Records in one dataset but not the other

If differences exist, the team meets to identify the root cause:
- Rounding rules different?
- Baseline definition different (first visit vs. screening)?
- Missing data handling different?

Once reconciled, the agreed-upon approach is coded and validated.

## 22.4 Unit Testing with testthat

**Unit testing** is a software development practice in which small, isolated code units are tested in isolation. In clinical programming, unit tests verify that:
- Data transformations work correctly.
- Edge cases (missing data, extreme values) are handled properly.
- Statistical calculations match known results.

### 22.4.1 testthat Package Basics

The **testthat** package provides a framework for writing tests:

```{r}

      SUBJID = c(1, 1, 2, 2),
      VISITNUM = c(1, 2, 1, 2),

    expect_equal(result$PCHG, expected$PCHG)
  })
}
```

## 22.8 Controlled R Environments with renv

### 22.8.1 Why renv Matters for Validation

Without renv, reproducibility is fragile:
- A colleague's system has a different R version.
- A new package update changes behavior.
- Six months later, you can't re-run an analysis.

**renv** solves this by "freezing" your project's R environment:

```
Project A (2024-01-15):
  R 4.3.2
  dplyr 1.1.0
  ggplot2 3.4.1

Project B (2023-06-20):
  R 4.2.0
  dplyr 1.0.10
  ggplot2 3.3.6
```

When you open Project A, renv automatically loads the frozen environment, ensuring reproducibility.

## 22.9 CDISC Compliance Checks

### 22.9.1 Pinnacle 21 (Conceptual Overview)

**Pinnacle 21 Community** is a web-based tool (by FDA/CDISC) that validates SDTM and ADaM datasets against CDISC standards. While not an R package, understanding its checks helps us validate in R.

Key Pinnacle 21 checks:
- **SDTM**: Variable names (STUDYID, SITEID, SUBJID), domains (DM, AE, LB), controlled terminology
- **ADaM**: Proper flags (ITT, EFFFL, SAFFL), analysis variables (AVAL, BASE, CHG)

### 22.9.2 Basic CDISC Compliance Checks in R

Create a function to validate ADaM variables:

```{r}
#| echo: true

# Validate CDISC ADaM structure
validate_adam <- function(df) {
  errors <- list()

  # Check required variables
  required_vars <- c("STUDYID", "USUBJID", "AVAL", "AVISIT")
  missing_vars <- setdiff(required_vars, names(df))
  if (length(missing_vars) > 0) {
    errors$missing_vars <- missing_vars
  }

  # Check no missing AVAL for analysis population
  if (any(is.na(df$AVAL))) {
    errors$missing_aval <- "AVAL contains missing values"
  }

  # Check AVISIT is controlled terminology
  valid_visits <- c("Baseline", "Week 2", "Week 4", "Week 8", "Week 12")
  invalid_visits <- setdiff(unique(df$AVISIT), valid_visits)
  if (length(invalid_visits) > 0) {
    errors$invalid_visits <- invalid_visits
  }

  # Return results
  if (length(errors) == 0) {
    cat("✓ ADaM dataset is CDISC-compliant\n")
    return(invisible(TRUE))
  } else {
    cat("✗ ADaM compliance errors found:\n")
    print(errors)
    return(invisible(FALSE))
  }
}

# Use the function (adlb_a created in section 22.3.2)
validate_adam(adlb_a)
```

## 22.11 Comprehensive Validation Workflow Example

Here's a complete, realistic validation workflow:

```{r}
#| echo: true
#| eval: false

# ============================================================================
# Validation Workflow: ADaM AE Analysis
# ============================================================================

library(dplyr)
if (requireNamespace("testthat", quietly = TRUE)) library(testthat)
if (requireNamespace("diffdf",   quietly = TRUE)) library(diffdf)
if (requireNamespace("logger",   quietly = TRUE)) library(logger)
library(renv)

# Setup logging
if (requireNamespace("logger", quietly = TRUE)) {
  log_appender(appender_file("adae_validation.log"))
  log_info("=== ADAE Validation Workflow ===")
  log_info("Date: %s", Sys.time())
}

# ============================================================================
# STEP 1: Load data and verify environment
# ============================================================================

if (requireNamespace("logger", quietly = TRUE)) {
  log_info("Step 1: Verifying R environment...")
  log_info("R version: %s", R.version$version.string)
  log_info("renv lockfile: %s", normalizePath("renv.lock"))
}

# Restore environment
renv::restore(confirm = FALSE)

# ============================================================================
# STEP 2: Load source data
# ============================================================================

if (requireNamespace("logger", quietly = TRUE)) log_info("Step 2: Loading SDTM data...")
ae <- read_csv("data/ae.csv")
if (requireNamespace("logger", quietly = TRUE)) log_info("Loaded AE domain: %d rows", nrow(ae))

# ============================================================================
# STEP 3: Programmer A's approach
# ============================================================================

if (requireNamespace("logger", quietly = TRUE)) log_info("Step 3a: Programmer A deriving ADAE...")

adae_prog_a <- ae %>%
  group_by(SUBJID, AESOC, AETERM) %>%
  mutate(
    AESEQ = row_number(),
    AESRDF = ifelse(AESER == "Y", "Serious", "Non-serious")
  ) %>%
  ungroup()

if (requireNamespace("logger", quietly = TRUE)) log_info("Programmer A ADAE: %d rows", nrow(adae_prog_a))

# ============================================================================
# STEP 4: Programmer B's approach (independent code)
# ============================================================================

if (requireNamespace("logger", quietly = TRUE)) log_info("Step 3b: Programmer B deriving ADAE...")

adae_prog_b <- ae %>%
  arrange(SUBJID, AESOC, AETERM) %>%
  group_by(SUBJID, AESOC, AETERM) %>%
  mutate(
    AESEQ = seq_along(SUBJID),
    AESRDF = case_when(
      AESER == "Y" ~ "Serious",
      AESER == "N" ~ "Non-serious",
      TRUE ~ NA_character_
    )
  ) %>%
  ungroup()

if (requireNamespace("logger", quietly = TRUE)) log_info("Programmer B ADAE: %d rows", nrow(adae_prog_b))

# ============================================================================
# STEP 5: Compare datasets
# ============================================================================

if (requireNamespace("logger", quietly = TRUE)) log_info("Step 4: Comparing ADAE outputs...")

if (requireNamespace("diffdf", quietly = TRUE)) {
  comparison <- diffdf(
    base = adae_prog_a,
    compare = adae_prog_b,
    keys = c("SUBJID", "AESOC", "AETERM", "AESEQ")
  )

  if (requireNamespace("logger", quietly = TRUE)) {
    if (is.null(comparison$comparison_result)) {
      log_info("✓ No differences: ADAE outputs are identical")
    } else {
      log_warn("✗ Differences detected:")
      log_warn(paste(capture.output(print(comparison)), collapse = "\n"))
    }
  }
}

# ============================================================================
# STEP 6: Unit tests
# ============================================================================

if (requireNamespace("logger",   quietly = TRUE)) log_info("Step 5: Running unit tests...")

if (requireNamespace("testthat", quietly = TRUE)) {
  test_that("AESRDF assigned correctly", {
    test_df <- data.frame(
      AESER = c("Y", "N", "Y"),
      expected = c("Serious", "Non-serious", "Serious")
    )

    actual <- ifelse(test_df$AESER == "Y", "Serious", "Non-serious")
    expect_equal(actual, test_df$expected)
  })
}

if (requireNamespace("logger", quietly = TRUE)) log_info("✓ All unit tests passed")

# ============================================================================
# STEP 7: Finalize ADAE
# ============================================================================

if (requireNamespace("logger", quietly = TRUE)) log_info("Step 6: Finalizing ADAE dataset...")

# Use agreed-upon approach after reconciliation
ADAE_FINAL <- adae_prog_a %>%
  select(STUDYID, SUBJID, AESEQ, AESOC, AETERM, AESEV, AESRDF, AEACN)

# Save
cat("ADAE_FINAL rows:", nrow(ADAE_FINAL), "(write_csv skipped in render)
")
if (requireNamespace("logger", quietly = TRUE)) log_info("ADAE saved: output/adae.csv (%d rows)", nrow(ADAE_FINAL))

# ============================================================================
# STEP 8: Summary
# ============================================================================

if (requireNamespace("logger", quietly = TRUE)) {
  log_info("=== Validation Workflow Complete ===")
  log_info("Status: PASSED")
  log_info("Timestamp: %s", Sys.time())
}
```

## 22.12 Checklist for Clinical Programming Validation

Use this checklist to ensure comprehensive validation:

- [ ] **Environment**: renv.lock created and tested
- [ ] **Unit Tests**: All core functions have testthat tests (≥90% coverage)
- [ ] **Double Programming**: Independent programmer reconciliation completed
- [ ] **Dataset Comparison**: diffdf comparison report generated
- [ ] **CDISC Compliance**: ADaM variables validated against standards
- [ ] **Logging**: All processing steps logged to audit trail
- [ ] **Documentation**: Validation plan and test scripts documented
- [ ] **Version Control**: Code and documentation committed to git
- [ ] **Sign-off**: Biostatistician and QA reviewer signatures on validation report

## 22.13 Troubleshooting Validation Issues

| Issue | Cause | Resolution |
|-------|-------|-----------|
| Test failures in CI/CD but pass locally | Different R or package versions | Use renv::restore() in CI |
| diffdf shows rounding differences | Different decimal places | Use tolerance: `tolerance = 0.001` |
| Logging generates huge files | Too much DEBUG-level logging | Lower log_level or filter by logger name |
| Unit tests fail on missing data | Test data doesn't include all edge cases | Add tests with NA, NaN, Inf |
| CDISC validation fails on terminology | Non-standard visit names | Use controlled terminology (CDISC library) |

## 22.14 Summary

- **Validation** in clinical programming ensures regulatory compliance and data integrity.
- **21 CFR Part 11** requires audit trails; version control and logging satisfy this.
- **Double programming** is gold standard; use diffdf to identify discrepancies.
- **Unit testing** with testthat provides evidence of correctness; aim for ≥90% coverage.
- **renv** freezes R environments, ensuring reproducibility across time and collaborators.
- **CDISC compliance** checks verify that ADaM datasets meet regulatory standards.
- **Logging** with logger or futile.logger creates permanent records of processing steps.
- Comprehensive **validation workflows** combine all these techniques into a cohesive system.
