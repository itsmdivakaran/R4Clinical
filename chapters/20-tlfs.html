31  Tables, Listings, and Figures (TLFs)

31.1 20.1 Introduction to Tables, Listings, and Figures

Clinical study reports (CSRs) submitted to regulatory agencies such as the FDA, EMA, and PMDA are built upon three foundational deliverables: Tables, Listings, and Figures (TLFs). These documents represent the core evidence of a drug’s safety and efficacy in the submitted indication.

A Table is a concise, summarized presentation of analysed data—for example, the primary efficacy endpoint results by treatment arm. A Listing is a detailed row-level display of individual subject data, typically used to support tables or present specific safety signals. A Figure is a visual representation of data, ranging from plots and graphs to forest plots and swimmer plots.

The stakes are high: regulatory reviewers scrutinize TLFs to assess the quality, completeness, and integrity of the trial. Poorly formatted tables, missing footnotes, inconsistent numbering, or lack of traceability can trigger Requests for Additional Information (RFIs), delaying approval by months or years. This chapter equips you with practical, standards-based workflows to generate publication-quality TLFs using R.

31.2 20.2 Regulatory Context and ICH E3 Structure

The ICH E3 guideline (“Structure and Content of Clinical Study Reports”) defines where TLFs appear in a CSR. Understanding this structure ensures your outputs align with regulatory expectations.

31.2.1 20.2.1 The ICH E3 Framework

A typical CSR structure follows:

  • Module A.2: Overview and quality overall summary (no TLFs)
  • Module B.3.2: Quality overall summary (no TLFs)
  • Module B.3.3: Nonclinical overview and summaries (limited TLFs)
  • Module B.3.4: Nonclinical study reports (TLFs from animal data)
  • Modules B.3.5–B.3.7: Clinical overview and summaries (key TLFs)
    • Demographics and baseline characteristics (Table 14.1.1)
    • Efficacy tables (Tables 14.3.x)
    • Safety tables (Tables 14.3.x)
    • Pharmacokinetics tables (if applicable)
  • Module 5.3: Clinical study reports (full TLFs and detailed analyses)

Table and figure numbering typically follows the pattern 14.X.Y (Module 5.3) or 14.3.X (summary sections). Listings are usually presented in appendices.

31.2.2 20.2.2 Population Flags in ADaM

Before generating TLFs, you must define the Analysis Populations (populations flags in ADaM datasets):

Population Abbreviation Definition Common Use
Intention-to-Treat ITT All randomized subjects Primary efficacy analysis
Per-Protocol PP Randomized subjects with ≥80% protocol adherence Sensitivity analysis
Safety SAF All dosed subjects Safety analyses
Pharmacokinetic PK Subjects with evaluable PK data PK/PD analyses

In ADaM, these are typically encoded as binary flags: ITT = "Y", PP = "Y", SAF = "Y", etc. When generating a Table 14.1.1 (Demographics), you filter to ITT = "Y" and group by treatment arm.

31.3 20.3 Table Standards and Formatting Rules

31.3.1 20.3.1 Core Formatting Principles

Clinical tables must adhere to several formatting rules:

  1. Decimal Alignment: Numeric columns align on the decimal point (right-aligned in most cases, but decimal-aligned in publication format).
  2. Header Rows: Include clear header rows with treatment arm names, N values, and percentage bases.
  3. Spanning Headers: Group related columns under a single header (e.g., “Treatment Arm A” spanning three columns: N, Mean, SD).
  4. Footnotes: Include superscript footnotes explaining special values, abbreviations, and analytical definitions.
  5. Page Numbering: Tables spanning multiple pages must include continued headers and page numbers.
  6. Abbreviations Table: A legend of all abbreviations used.

31.4 20.4 Key R Packages for Clinical Tables

31.4.1 20.4.1 gtsummary: Quick Demographic Tables

The gtsummary package excels at producing publication-ready demographic and baseline tables with minimal code.

Key functions: - tbl_summary(): Creates a demographic table from a dataframe. - tbl_regression(): Produces regression tables. - add_stat(), add_p(): Add custom statistics and p-values. - as_gt(), as_flextable(): Convert to other table formats.

31.4.2 20.4.2 gt: Highly Customizable Tables

gt (grammar of tables) provides granular control over table styling, colors, fonts, and layout. It excels for complex, multi-level headers and custom cell formatting.

31.4.3 20.4.3 flextable: Word/RTF-Compatible Output

flextable is ideal when you need to export to Word (.docx) or RTF—common regulatory deliverable formats. It supports: - Merged cells and spanning headers - Conditional formatting (cell coloring) - Page breaks between rows - Native Word integration

31.4.4 20.4.4 rtables: Roche’s Framework (Brief Intro)

rtables is a specialized framework developed by Roche for clinical trial reporting. It enforces a row-column structure optimized for regulatory tables. While powerful, it has a steeper learning curve; we’ll focus on gtsummary, gt, and flextable in this chapter.

demographics_table <- dm %>%

    by = ARM,
    label = list(
      AGE ~ "Age (years)",

      RACE ~ "Race"

      AGE ~ "continuous",
      SEX ~ "categorical",

      all_continuous() ~ "{mean} ({sd})",
      all_categorical() ~ "{n} ({p}%)"
    )
  ) %>%
  add_overall() %>%
  modify_header(label = "Characteristic") %>%
  modify_caption("**Table 14.1.1:** Demographics and Baseline Characteristics – Intent-to-Treat Population")

# Count AEs by SOC and PT, by treatment arm
ae_summary <- ae %>%
  left_join(dm %>% filter(ARM != "Screen Failure") %>% select(USUBJID, ARM), by = "USUBJID") %>%
  filter(!is.na(ARM)) %>%
  distinct(USUBJID, AESOC, AETERM, ARM) %>%
  group_by(ARM, AESOC, AETERM) %>%
  summarise(n = n(), .groups = 'drop') %>%
  # Get total N per arm
  left_join(
    dm %>%
      filter(ARM != "Screen Failure") %>%
      group_by(ARM) %>%
      summarise(total_n = n(), .groups = 'drop'),
    by = "ARM"
  ) %>%
  mutate(pct = round(100 * n / total_n, 1)) %>%
  arrange(AESOC, AETERM)

31.7.2 20.7.2 Example: Kaplan-Meier Survival Curve

library(survival)
library(ggplot2)

# Simulate survival data
set.seed(42)
survival_data <- data.frame(
  SUBJID = 1:300,
  ARM = rep(c("Treatment A", "Treatment B", "Control"), each = 100),
  time = c(
    rexp(100, rate = 0.03),  # Treatment A: longer survival
    rexp(100, rate = 0.04),  # Treatment B: intermediate
    rexp(100, rate = 0.05)   # Control: shorter survival
  ),
  event = rep(c(rep(1, 80), rep(0, 20)), 3)  # 80% events, 20% censored
)

# Fit survival curves
survfit_obj <- survfit(Surv(time, event) ~ ARM, data = survival_data)

# KM plot using base survival + ggplot2 (no ggsurvfit dependency)
km_sum <- summary(survfit_obj)
km_df  <- data.frame(
  time   = km_sum$time,
  surv   = km_sum$surv,
  lower  = km_sum$lower,
  upper  = km_sum$upper,
  ARM    = sub("ARM=", "", as.character(km_sum$strata))
)

km_plot <- ggplot(km_df, aes(x = time, y = surv, colour = ARM, fill = ARM)) +
  geom_step(linewidth = 0.9) +
  geom_ribbon(aes(ymin = lower, ymax = upper), alpha = 0.15, colour = NA) +
  scale_y_continuous(labels = scales::percent, limits = c(0, 1)) +
  scale_colour_manual(values = c("#2C6E9E", "#E07B39", "#4CAF50")) +
  scale_fill_manual(  values = c("#2C6E9E", "#E07B39", "#4CAF50")) +
  labs(
    title    = "Figure 14.2.1: Kaplan-Meier Survival Curve",
    subtitle = "Intent-to-Treat Population",
    x        = "Time (days)",
    y        = "Probability of Survival",
    colour   = "Treatment",
    fill     = "Treatment"
  ) +
  theme_bw(base_size = 11) +
  theme(
    plot.title       = element_text(face = "bold", size = 13),
    legend.position  = "bottom",
    panel.grid.minor = element_blank()
  )

# Tables, Listings, and Figures (TLFs)


## 20.1 Introduction to Tables, Listings, and Figures

Clinical study reports (CSRs) submitted to regulatory agencies such as the FDA, EMA, and PMDA are built upon three foundational deliverables: **Tables**, **Listings**, and **Figures** (TLFs). These documents represent the core evidence of a drug's safety and efficacy in the submitted indication.

A Table is a concise, summarized presentation of analysed data—for example, the primary efficacy endpoint results by treatment arm. A Listing is a detailed row-level display of individual subject data, typically used to support tables or present specific safety signals. A Figure is a visual representation of data, ranging from plots and graphs to forest plots and swimmer plots.

The stakes are high: regulatory reviewers scrutinize TLFs to assess the quality, completeness, and integrity of the trial. Poorly formatted tables, missing footnotes, inconsistent numbering, or lack of traceability can trigger Requests for Additional Information (RFIs), delaying approval by months or years. This chapter equips you with practical, standards-based workflows to generate publication-quality TLFs using R.

## 20.2 Regulatory Context and ICH E3 Structure

The ICH E3 guideline ("Structure and Content of Clinical Study Reports") defines where TLFs appear in a CSR. Understanding this structure ensures your outputs align with regulatory expectations.

### 20.2.1 The ICH E3 Framework

A typical **CSR structure** follows:

- **Module A.2**: Overview and quality overall summary (no TLFs)
- **Module B.3.2**: Quality overall summary (no TLFs)
- **Module B.3.3**: Nonclinical overview and summaries (limited TLFs)
- **Module B.3.4**: Nonclinical study reports (TLFs from animal data)
- **Modules B.3.5–B.3.7**: Clinical overview and summaries (key TLFs)
  - Demographics and baseline characteristics (Table 14.1.1)
  - Efficacy tables (Tables 14.3.x)
  - Safety tables (Tables 14.3.x)
  - Pharmacokinetics tables (if applicable)
- **Module 5.3**: Clinical study reports (full TLFs and detailed analyses)

Table and figure numbering typically follows the pattern **14.X.Y** (Module 5.3) or **14.3.X** (summary sections). Listings are usually presented in appendices.

### 20.2.2 Population Flags in ADaM

Before generating TLFs, you must define the **Analysis Populations** (populations flags in ADaM datasets):

| Population | Abbreviation | Definition | Common Use |
|-----------|--------------|-----------|-----------|
| Intention-to-Treat | ITT | All randomized subjects | Primary efficacy analysis |
| Per-Protocol | PP | Randomized subjects with ≥80% protocol adherence | Sensitivity analysis |
| Safety | SAF | All dosed subjects | Safety analyses |
| Pharmacokinetic | PK | Subjects with evaluable PK data | PK/PD analyses |

In ADaM, these are typically encoded as binary flags: `ITT = "Y"`, `PP = "Y"`, `SAF = "Y"`, etc. When generating a Table 14.1.1 (Demographics), you filter to `ITT = "Y"` and group by treatment arm.

## 20.3 Table Standards and Formatting Rules

### 20.3.1 Core Formatting Principles

Clinical tables must adhere to several formatting rules:

1. **Decimal Alignment**: Numeric columns align on the decimal point (right-aligned in most cases, but decimal-aligned in publication format).
2. **Header Rows**: Include clear header rows with treatment arm names, N values, and percentage bases.
3. **Spanning Headers**: Group related columns under a single header (e.g., "Treatment Arm A" spanning three columns: N, Mean, SD).
4. **Footnotes**: Include superscript footnotes explaining special values, abbreviations, and analytical definitions.
5. **Page Numbering**: Tables spanning multiple pages must include continued headers and page numbers.
6. **Abbreviations Table**: A legend of all abbreviations used.

## 20.4 Key R Packages for Clinical Tables

### 20.4.1 gtsummary: Quick Demographic Tables

The **gtsummary** package excels at producing publication-ready demographic and baseline tables with minimal code.

**Key functions**:
- `tbl_summary()`: Creates a demographic table from a dataframe.
- `tbl_regression()`: Produces regression tables.
- `add_stat()`, `add_p()`: Add custom statistics and p-values.
- `as_gt()`, `as_flextable()`: Convert to other table formats.

### 20.4.2 gt: Highly Customizable Tables

**gt** (grammar of tables) provides granular control over table styling, colors, fonts, and layout. It excels for complex, multi-level headers and custom cell formatting.

### 20.4.3 flextable: Word/RTF-Compatible Output

**flextable** is ideal when you need to export to Word (.docx) or RTF—common regulatory deliverable formats. It supports:
- Merged cells and spanning headers
- Conditional formatting (cell coloring)
- Page breaks between rows
- Native Word integration

### 20.4.4 rtables: Roche's Framework (Brief Intro)

**rtables** is a specialized framework developed by Roche for clinical trial reporting. It enforces a row-column structure optimized for regulatory tables. While powerful, it has a steeper learning curve; we'll focus on gtsummary, gt, and flextable in this chapter.

## 20.7 Figures for Regulatory Submission

### 20.7.1 Resolution and Quality Standards

Regulatory figures must meet strict technical requirements:

- **Resolution**: ≥300 DPI (dots per inch) for print publication.
- **Font Size**: Minimum 10 pt (Arial or Helvetica preferred).
- **Color**: Use colorblind-friendly palettes (e.g., viridis); provide grayscale alternatives.
- **Labels**: All axes, legends, and footnotes must be clearly labeled.
- **Format**: PDF is preferred for regulatory submissions (preserves formatting).

### 20.7.2 Example: Kaplan-Meier Survival Curve

set.seed(42)

## 20.9 Troubleshooting Common TLF Issues

| Issue | Cause | Solution |

| Spanning headers not aligned | Incorrect `colwidths` in `add_header_row()` | Ensure `colwidths` sum equals number of columns |
