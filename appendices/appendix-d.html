41.4 Estimand Framework: ICH E9(R1) Deep Dive

Understanding estimands is critical for pre-specification and sensitivity analysis.

41.4.1 The Five Estimand Attributes

Every estimand has these five mandatory components:

1. Treatment Definition (Population-level treatment)

  • What drug, dose, regimen?
  • When does treatment start/stop?
  • Include treatment modifications?

Example: “Single dose of 100mg Drug_A given orally on Day 1”

2. Population (Target patient population)

  • Inclusion/exclusion criteria
  • Stratification factors (e.g., disease severity, age)

Example: “Patients aged 18-75 years with mild-to-moderate hypertension”

3. Endpoint (Variable/outcome being measured)

  • Which outcome? (primary vs. secondary)
  • Timing (e.g., Week 12, change from baseline)
  • How is it measured?

Example: “Change in systolic blood pressure from baseline to Week 12”

4. Intercurrent Events Strategy (How to handle interruptions)

Define a priori how to handle events that occur during the trial:

Strategy Definition When Used Example
Treatment Policy Measure outcome regardless of strategy; include post-intervention data When compliance is not the estimand ITT analysis; all subjects followed regardless of dose changes
Hypothetical Estimate outcome had strategy not occurred; exclude post-event data When counterfactual is desired “What if no one discontinued?”
Composite Treat strategy-related event as part of outcome When event is clinically relevant Time to death or disease progression
While-on-Treatment Censor at time of strategy occurrence When only on-treatment effect is of interest Safety during exposure
Principal Stratum Restrict population to predicted non-occurrers of strategy When selection matters Efficacy in compliant subjects

Example for hypertension trial: “For subjects who discontinue study drug, continue measuring BP per protocol (treatment policy strategy).”

5. Summary Measure (How to synthesize across population)

  • Point estimate (mean, median)?
  • Risk ratio, odds ratio, hazard ratio?
  • One-stage vs. two-stage analysis?

Example: “Mean difference in SBP change between arms”

41.4.2 Operationalizing Estimands in R

# Example: Estimand specification
estimand_spec <- list(
  treatment_definition = "100mg once daily for 12 weeks",
  population = "HTN patients aged 18-75",
  endpoint = "Change from baseline SBP at Week 12",
  intercurrent_event = "Treatment policy (include all subjects)",
  summary_measure = "Mean difference between arms"
)

# Primary analysis (aligns with estimand)
primary_fit <- lm(sbp_change_w12 ~ arm, data = analysis_data)

# Sensitivity 1: Per-protocol population (hypothetical strategy)
pp_fit <- lm(sbp_change_w12 ~ arm, data = analysis_data[protocol_adherence == "Yes", ])

# Sensitivity 2: As-treated (while-on-treatment strategy)
aot_fit <- lm(sbp_change_w12 ~ arm, data = analysis_data[!is.na(days_on_drug), ])

41.8 R vs. SAS in Regulatory Context

A comparison for organizations transitioning from SAS to R.

Aspect SAS R Regulatory Status
FDA Acceptance Established (40+ years) Accepted (2021+) Both approved for submissions
Validation 21 CFR Part 11 documentation riskmetric, valtools packages Both acceptable with proper documentation
Package Governance Single vendor (SAS Institute) Multiple (CRAN, GitHub, pharmaverse) Pharmaverse packages FDA-vetted
Documentation Company-controlled Open-source, version-controlled R transparency advantage
Community Industry-focused Growing academic/industry mix Both have strong communities
Cost High licensing Free (open-source) R advantageous for cost
Learning Curve Steep; data-step paradigm Moderate; functional programming R easier for new programmers
XPT Export PROC XCOPY (complex) xportr::write_xpt() (simple) R cleaner interface
CDISC Compliance Manual verification Automated (admiral framework) R more efficient

Recommendation: Organizations should use R for new projects; SAS for legacy system maintenance. Parallel validation initially (SAS → R comparison).

41.9 Validation and Quality Assurance in R

41.9.1 Validation Framework (21 CFR Part 11)

For regulatory submissions, R code and output must be traceable, reproducible, and validated.

Key Requirements

  1. User identification and authentication
    • Access controls, audit logs
    • GitHub or version control system
  2. System validation
    • riskmetric package: assess package quality
    • valtools: generate validation reports
  3. Data integrity
    • Version control (Git)
    • Checksums/hashes for datasets
    • diffdf for before/after comparison
  4. Audit trail
    • Quarto/RMarkdown with session_info()
    • renv lock file for reproducibility
    • Git commits with timestamps

R Validation Tools

library(riskmetric)

# Assess package quality for regulatory use
pkg_assessment <- pkg_assess(pkg_reference("gtsummary"))
assess_score <- mean_assess(pkg_assessment)

# Output validation metrics
print(pkg_assessment)

# Use renv for reproducible environments
renv::init()          # Initialize project
renv::snapshot()      # Lock package versions
renv::restore()       # Restore exact versions

41.10 Summary: Regulatory Compliance Checklist

When submitting clinical analyses to FDA:

Pre-Analysis (Design Phase) - [ ] Estimand specification (ICH E9(R1)) documented - [ ] Statistical analysis plan (SAP) approved - [ ] Sample size justification - [ ] Multiple testing adjustment strategy

Data Phase - [ ] SDTM datasets created per CDISC v3.4 - [ ] Define-XML metadata complete - [ ] Data validation and quality checks documented - [ ] Missing data mechanisms understood

Analysis Phase - [ ] ADaM datasets derived using admiral framework - [ ] Analysis code version-controlled (GitHub) - [ ] Sensitivity analyses per estimand strategy - [ ] Results tables per tern/gtsummary standards

Reporting Phase - [ ] XPT files created with xportr - [ ] Clinical Study Report (CSR) structure aligned with ICH E3 - [ ] SAS transport files validated - [ ] Validation summary documenting R version, package versions (renv)

Submission Phase - [ ] R submission package contents (code, data, outputs) - [ ] Traceability matrix (specifications ↔︎ code ↔︎ outputs) - [ ] Validation report (riskmetric assessment) - [ ] Regulatory letters of support (R Consortium examples)

## Estimand Framework: ICH E9(R1) Deep Dive

Understanding estimands is critical for pre-specification and sensitivity analysis.

### The Five Estimand Attributes

Every estimand has these five mandatory components:

#### 1. **Treatment Definition** (Population-level treatment)
- What drug, dose, regimen?
- When does treatment start/stop?
- Include treatment modifications?

**Example**: "Single dose of 100mg Drug_A given orally on Day 1"

#### 2. **Population** (Target patient population)
- Inclusion/exclusion criteria
- Stratification factors (e.g., disease severity, age)

**Example**: "Patients aged 18-75 years with mild-to-moderate hypertension"

#### 3. **Endpoint** (Variable/outcome being measured)
- Which outcome? (primary vs. secondary)
- Timing (e.g., Week 12, change from baseline)
- How is it measured?

**Example**: "Change in systolic blood pressure from baseline to Week 12"

#### 4. **Intercurrent Events Strategy** (How to handle interruptions)
Define a priori how to handle events that occur during the trial:

| Strategy | Definition | When Used | Example |
|----------|-----------|-----------|---------|
| **Treatment Policy** | Measure outcome regardless of strategy; include post-intervention data | When compliance is not the estimand | ITT analysis; all subjects followed regardless of dose changes |
| **Hypothetical** | Estimate outcome had strategy not occurred; exclude post-event data | When counterfactual is desired | "What if no one discontinued?" |
| **Composite** | Treat strategy-related event as part of outcome | When event is clinically relevant | Time to death or disease progression |
| **While-on-Treatment** | Censor at time of strategy occurrence | When only on-treatment effect is of interest | Safety during exposure |
| **Principal Stratum** | Restrict population to predicted non-occurrers of strategy | When selection matters | Efficacy in compliant subjects |

**Example for hypertension trial:**
"For subjects who discontinue study drug, continue measuring BP per protocol (treatment policy strategy)."

#### 5. **Summary Measure** (How to synthesize across population)
- Point estimate (mean, median)?
- Risk ratio, odds ratio, hazard ratio?
- One-stage vs. two-stage analysis?

**Example**: "Mean difference in SBP change between arms"

### Operationalizing Estimands in R

```r
# Example: Estimand specification
estimand_spec <- list(
  treatment_definition = "100mg once daily for 12 weeks",
  population = "HTN patients aged 18-75",
  endpoint = "Change from baseline SBP at Week 12",
  intercurrent_event = "Treatment policy (include all subjects)",
  summary_measure = "Mean difference between arms"
)

# Primary analysis (aligns with estimand)
primary_fit <- lm(sbp_change_w12 ~ arm, data = analysis_data)

# Sensitivity 1: Per-protocol population (hypothetical strategy)
pp_fit <- lm(sbp_change_w12 ~ arm, data = analysis_data[protocol_adherence == "Yes", ])

# Sensitivity 2: As-treated (while-on-treatment strategy)
aot_fit <- lm(sbp_change_w12 ~ arm, data = analysis_data[!is.na(days_on_drug), ])
```

## R vs. SAS in Regulatory Context

A comparison for organizations transitioning from SAS to R.

| Aspect | SAS | R | Regulatory Status |
|--------|-----|---|------------------|
| **FDA Acceptance** | Established (40+ years) | Accepted (2021+) | Both approved for submissions |
| **Validation** | 21 CFR Part 11 documentation | riskmetric, valtools packages | Both acceptable with proper documentation |
| **Package Governance** | Single vendor (SAS Institute) | Multiple (CRAN, GitHub, pharmaverse) | Pharmaverse packages FDA-vetted |
| **Documentation** | Company-controlled | Open-source, version-controlled | R transparency advantage |
| **Community** | Industry-focused | Growing academic/industry mix | Both have strong communities |
| **Cost** | High licensing | Free (open-source) | R advantageous for cost |
| **Learning Curve** | Steep; data-step paradigm | Moderate; functional programming | R easier for new programmers |
| **XPT Export** | `PROC XCOPY` (complex) | `xportr::write_xpt()` (simple) | R cleaner interface |
| **CDISC Compliance** | Manual verification | Automated (admiral framework) | R more efficient |

**Recommendation**: Organizations should use R for new projects; SAS for legacy system maintenance. Parallel validation initially (SAS → R comparison).

## Validation and Quality Assurance in R

### Validation Framework (21 CFR Part 11)

For regulatory submissions, R code and output must be traceable, reproducible, and validated.

#### Key Requirements

1. **User identification and authentication**
   - Access controls, audit logs
   - GitHub or version control system

2. **System validation**
   - `riskmetric` package: assess package quality
   - `valtools`: generate validation reports

3. **Data integrity**
   - Version control (Git)
   - Checksums/hashes for datasets
   - `diffdf` for before/after comparison

4. **Audit trail**
   - Quarto/RMarkdown with `session_info()`
   - `renv` lock file for reproducibility
   - Git commits with timestamps

#### R Validation Tools

```r
library(riskmetric)

# Assess package quality for regulatory use
pkg_assessment <- pkg_assess(pkg_reference("gtsummary"))
assess_score <- mean_assess(pkg_assessment)

# Output validation metrics
print(pkg_assessment)

# Use renv for reproducible environments
renv::init()          # Initialize project
renv::snapshot()      # Lock package versions
renv::restore()       # Restore exact versions
```

## Summary: Regulatory Compliance Checklist

When submitting clinical analyses to FDA:

**Pre-Analysis (Design Phase)**
- [ ] Estimand specification (ICH E9(R1)) documented
- [ ] Statistical analysis plan (SAP) approved
- [ ] Sample size justification
- [ ] Multiple testing adjustment strategy

**Data Phase**
- [ ] SDTM datasets created per CDISC v3.4
- [ ] Define-XML metadata complete
- [ ] Data validation and quality checks documented
- [ ] Missing data mechanisms understood

**Analysis Phase**
- [ ] ADaM datasets derived using admiral framework
- [ ] Analysis code version-controlled (GitHub)
- [ ] Sensitivity analyses per estimand strategy
- [ ] Results tables per tern/gtsummary standards

**Reporting Phase**
- [ ] XPT files created with xportr
- [ ] Clinical Study Report (CSR) structure aligned with ICH E3
- [ ] SAS transport files validated
- [ ] Validation summary documenting R version, package versions (renv)

**Submission Phase**
- [ ] R submission package contents (code, data, outputs)
- [ ] Traceability matrix (specifications ↔ code ↔ outputs)
- [ ] Validation report (riskmetric assessment)
- [ ] Regulatory letters of support (R Consortium examples)
