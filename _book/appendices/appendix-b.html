39 Appendix B: Essential R Packages for Clinical Data Analysis

39.1 Introduction: The Pharmaverse Ecosystem

The pharmaverse is a curated collection of R packages designed specifically for pharmaceutical research and clinical data analysis. It represents the gold standard for regulatory-compliant statistical computing, with packages built and maintained by FDA-engaged statisticians and industry leaders.

Key principles of the pharmaverse: - CDISC-compliant: Supports SDTM, ADaM, and CDASH standards - Regulatory-ready: Used in FDA submissions since 2021 - Open-source: Community-driven development - Interoperable: Packages work seamlessly together - Well-documented: Extensive vignettes and examples

Official resources: - Website: https://pharmaverse.org - GitHub: https://github.com/pharmaverse - Community: Slack, issues, discussions

39.2 Pharmaverse Core Packages

These packages form the foundation for clinical data standardization and transformation.

Package Latest Version Purpose Key Functions When to Use
admiral 1.1+ CDISC ADaM dataset derivations derive_vars_*(), derive_params_*(), create_*() Building ADaM-compliant analysis datasets from SDTM
admiraldev 0.1+ Extensions for special therapeutic areas (e.g., oncology, cardiovascular) Oncology-specific: derive_param_*() for response, time-to-event Specialized therapeutic domains beyond base admiral

pharmaverseadam 0.2+ ADaM template datasets Example ADaM structures (ADSL, ADAE, ADCM) Reference implementations and unit testing

sdtm.oak 0.1+ Automating SDTM derivations oak_derive_var_*(), oak_auto_assign() Reducing manual SDTM programming

Installation:

# Install pharmaverse packages (individual)
install.packages("admiral")
install.packages("xportr")

# Or install from GitHub for development versions
remotes::install_github("pharmaverse/admiral")
remotes::install_github("pharmaverse/pharmaversesdtm")

trial_data %>%

39.5 Survival Analysis Packages

Specialized packages for time-to-event analyses.

Package Latest Version Purpose Key Functions When to Use

survminer 0.4+ Survival visualization ggsurvplot(), arrange_ggsurvplots(), ggcoxdiagnostics() Publication-ready KM plots with at-risk tables

Example: Cox regression and KM plot:

library(survival)
library(ggsurvfit)

# Create survival object
lung_data <- lung %>%
  mutate(time_years = time / 365.25)

# Fit Cox model
cox_model <- coxph(Surv(time_years, status) ~ age + sex + wt.loss,
                    data = lung_data)
summary(cox_model)

# KM curve by sex
library(ggsurvfit)
survfit(Surv(time_years, status) ~ sex, data = lung_data) %>%
  ggsurvfit() +
  labs(title = "Overall Survival by Sex",
       x = "Time (years)",
       y = "Overall Survival Probability")

MMRM example (recommended for clinical trials):

library(mmrm)

# Fit MMRM: primary analysis for longitudinal efficacy
mmrm_fit <- mmrm(
  formula = FEV1 ~ AVISIT * ARMCD + ar1(AVISIT | USUBJID),
  data = fev1_data,
  reml = TRUE
)

# Extract estimates
summary(mmrm_fit)

# Contrasts at endpoint (Visit 12)
library(emmeans)
emmeans(mmrm_fit, ~ ARMCD | AVISIT) %>%
  contrast(method = "trt.vs.ctrlk")

Multiple imputation workflow:

library(mice)

# Create 5 imputed datasets using MICE
imputed <- mice(clinical_data, m = 5, method = "pmm", seed = 123)

# Fit model to each imputed dataset
fit <- with(imputed, lm(outcome ~ treatment + age + sex))

# Pool estimates using Rubin's rules
pooled <- pool(fit)
summary(pooled)

# Get imputed datasets for manual analysis
complete_data <- complete(imputed, action = "long")

Example: Baseline characteristics table:

library(gtsummary)

baseline_tbl <- trial_data %>%
  select(arm, age, sex, stage, marker) %>%
  tbl_summary(
    by = arm,
    type = list(age ~ "continuous", marker ~ "continuous"),
    statistic = list(
      all_continuous() ~ "{mean} ({sd})",
      all_categorical() ~ "{n} ({p}%)"
    ),
    label = list(age = "Age (years)", marker = "Biomarker Level")
  ) %>%
  add_overall() %>%
  add_p(test = everything() ~ "t.test") %>%
  add_q() %>%
  modify_spanning_header(everything() ~ "**Treatment Arm**")

# Export to Word
gtsave(baseline_tbl, "baseline_table.docx")

# Or use flextable for more control
library(flextable)
ft <- as_flextable(baseline_tbl)
save_as_rtf(ft, path = "baseline_table.rtf")

Example: Bayesian logistic regression:

library(rstanarm)

# Bayesian logistic regression with default priors
bayes_fit <- stan_glm(
  response ~ age + sex + treatment,
  family = binomial(link = "logit"),
  data = trial_data,
  cores = 4,
  seed = 123
)

# Posterior summary
summary(bayes_fit)

# Posterior predictive check
library(bayesplot)
pp_check(bayes_fit)

# Hypothesis: treatment effect > 0
hypothesis(bayes_fit, "treatmentDrug_A > 0")

library(MatchIt)

# Fit propensity score model and match
m_out <- matchit(
  treatment ~ age + sex + comorbidity,
  data = observational_data,
  method = "nearest",
  distance = "glm"
)

# Extract matched data
matched_data <- get_matches(m_out)

# Check balance
library(cobalt)
love.plot(m_out, threshold = 0.1)

# Analyze matched sample
library(survey)
design <- svydesign(~1, weights = ~weights, data = matched_data)
svyglm(outcome ~ treatment, design = design)

39.16 Version Management with renv

For regulatory compliance and reproducibility, always use renv to manage package versions:

# Initialize renv in your project
renv::init()

# Develop and install packages as needed
install.packages("dplyr")

# Create snapshot of current environment
renv::snapshot()  # Creates renv.lock

# Later (or by others), restore exact same versions
renv::restore()  # Reads renv.lock

The renv.lock file should be version-controlled (Git) to ensure everyone has identical environments.

## Version Management with renv

For regulatory compliance and reproducibility, always use `renv` to manage package versions:

```r
# Initialize renv in your project
renv::init()

# Develop and install packages as needed
install.packages("dplyr")

# Create snapshot of current environment
renv::snapshot()  # Creates renv.lock

# Later (or by others), restore exact same versions
renv::restore()  # Reads renv.lock
```

The `renv.lock` file should be version-controlled (Git) to ensure everyone has identical environments.
