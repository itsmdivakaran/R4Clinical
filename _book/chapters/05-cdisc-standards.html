7  Chapter 5: CDISC Standards (SDTM & ADaM)

7.1 Learning Objectives

By the end of this chapter, you will: - Understand why CDISC standards are critical for regulatory submissions - Navigate SDTM domain structures and naming conventions - Recognize key ADaM datasets and their relationships to SDTM - Build and derive analysis datasets from SDTM domains - Implement metadata and traceability in clinical data workflows - Use R packages (pharmaversesdtm, admiral) to work with standards-compliant data

7.2 5.1 Why CDISC Standards Matter

7.2.1 The Regulatory Imperative

When Moderna submitted data for the COVID-19 vaccine trial to the FDA in late 2020, every table, figure, and listing (TFL) submitted was built from standardized data formatted according to CDISC standards. The same was true for the Pfizer-BioNTech submission. This isn’t an accident or best practice—it’s a regulatory requirement.

The Clinical Data Interchange Standards Consortium (CDISC) publishes standards that:

  1. Ensure regulatory acceptance: FDA, EMA, PMDA, and Health Canada accept CDISC-formatted data
  2. Enable data sharing: Multiple sites, CROs, and statisticians work with consistent definitions
  3. Facilitate audits: Regulators and auditors can trace any analysis result back to source data
  4. Reduce transcription errors: Automated validation catches inconsistencies
  5. Enable re-use: Future analyses (meta-analyses, real-world studies) can leverage existing data
ImportantFDA Guidance

The FDA’s guidance on “Providing Regulatory Submissions in Electronic Format—Certain Human and Animal Drug Applications” (2013, updated 2018) explicitly recommends CDISC standards for submissions. Many recent submissions now use CDISC as a requirement, not a suggestion.

7.2.2 Real-World Context: A Hypertension Trial

Imagine you’re a statistician at a 12-week randomized controlled trial comparing three antihypertensive drugs (treatment arms A, B, C) in 300 patients across 15 sites. Data comes from:

  • Electronic Case Report Forms (eCRFs) at each site
  • Laboratory systems (automated uploads of lab values)
  • Vital sign machines (automated BP readings)
  • Pharmacy systems (dosing and medication history)
  • Safety reporting forms (adverse events)

Without standardization, merging these sources would be chaotic. CDISC solves this by providing: - A common vocabulary (what does “baseline” mean? Visit 1? First post-dose? The median of 3 pre-dose readings?) - Consistent variable names (SYSTOLIC vs SBP vs SBPRES?) - Traceable data flow (SDTM → ADaM → TLF)

7.3 5.2 SDTM: Study Data Tabulation Model

7.3.1 Overview

The SDTM is the FDA-preferred format for submitting raw and derived clinical data. Think of SDTM as the “source of truth”—it documents exactly what happened in the trial, with minimal interpretation.

Key principles: - One observation per row: Each row represents one test, one adverse event, one medication administration, etc. - Domains: Data is organized by type (demographics, adverse events, labs, vital signs) - Standardized variable naming: Variables follow consistent patterns - No derived calculations: Values are directly from the case report form (CRF) or automatically captured systems - Traceability: Each value can be traced back to its source

7.3.2 SDTM Domains

Major domains include:

Domain Full Name Content Rows
DM Demographics Age, sex, ethnicity, enrollment date One per subject
AE Adverse Events All reported adverse events One per event
LB Laboratory All lab test results One per test per visit
VS Vital Signs Blood pressure, heart rate, temperature One per type per visit
EX Exposure Drug doses, administration times One per dose per day
CM Concomitant Medications Other drugs taken during trial One per medication per date range
MH Medical History Pre-trial medical conditions One per condition
SV Subject Visits Planned visits One per visit
PE Physical Examination Exam findings One per exam

For our hypertension example, key domains are DM (baseline characteristics), VS (blood pressure readings), LB (kidney function, potassium), AE (side effects), EX (drug doses), and CM (background medications like other antihypertensives).

7.3.3 SDTM Variable Naming Conventions

SDTM uses a systematic naming structure:

Prefix–Suffix pattern: - STUDYID: Study identifier (e.g., “HT-2024-001”) - USUBJID: Unique subject identifier (e.g., “HT-2024-001-0042”) - –STDTC / –DTC: Start/occurrence date-time in ISO 8601 format (YYYY-MM-DD’T’HH:MM:SS) - Example: “2024-03-15” or “2024-03-15T09:30:00” - –DTCF: Date imputation flag (shows if date was complete or imputed) - DOMAIN: Two-letter domain code (DM, AE, LB, etc.) - –SEQ: Sequence number (row ID within domain for a subject) - –TESTCD: Short test code (e.g., “SYSBP” for systolic BP) - –TEST: Long test name (e.g., “Systolic Blood Pressure”) - –RESULT: The actual value - –UNIT: Unit of measurement (e.g., “mmHg”, “g/dL”) - –VISIT: Visit name (e.g., “Week 0”, “Week 4”) - –VISITNUM: Visit sequence (0 = baseline, 1 = week 2, etc.)

Example: In the VS domain for systolic blood pressure: - VSDTC = “2024-03-15T09:30:00” (date/time of measurement — ISO 8601) - VSTEST = “Systolic Blood Pressure” (full test name) - VSTESTCD = “SYSBP” (short code) - VSSTRESN = 148 (standardised numeric result) - VSSTRESU = “mmHg” (standardised unit)

This structure ensures any analyst can understand the data without additional documentation.

TipCDISC Controlled Terminology

The CDISC Controlled Terminology (CT) defines allowed values for many variables. For example, SEXN (sex numeric) uses 1 = Male, 2 = Female. The CT is updated annually; always reference the correct version used in your trial protocol.

7.4 5.3 ADaM: Analysis Data Model

7.4.1 Overview and Relationship to SDTM

While SDTM captures raw data, analysis requires derived variables. This is where ADaM comes in.

ADaM (Analysis Data Model) is built from SDTM but adds: - Derived variables: Calculated fields like change from baseline, treatment-emergent status - Subject-level data: Demographics merged with outcomes - Analysis-ready format: One row = one analysis unit (subject, subject-visit, subject-event) - Audit trail: Variables show which SDTM records contributed to each ADaM row

The SDTM → ADaM → TLF flow:

Raw Data (CRF, Labs, EHR)
        ↓
SDTM (standardized raw data, one observation per event)
        ↓
ADaM (derived analysis data, one row = one analysis unit)
        ↓
TLF (Tables, Listings, Figures for submission)

7.4.2 Key ADaM Datasets

1. ADSL: Subject-Level Analysis Dataset - One row per subject - Contains: demographics, baseline characteristics, treatment assignment, trial status - Example variables: - USUBJID, STUDYID - TRT01A (actual treatment assigned) - TRT01PN (treatment sequence number) - TRTSTDT (treatment start date) - TRTENDT (treatment end date) - SAFFL (safety population flag, 1 if in analysis) - EFFFL (efficacy population flag) - DTHFL (death flag) / RFENDTC (end of reference period - used to derive discontinuation) - BASE (baseline value of primary outcome) - AGE, SEX, ETHNIC, RACE

2. ADAE: Adverse Events Analysis Dataset - One row per adverse event per subject (or one row per event for detailed analysis) - Built from SDTM AE domain - Added variables: - AESEV (severity: mild, moderate, severe) - AEREL (relationship to study drug: unrelated, remote, possible, probable, definite) - AESER (serious adverse event flag) - AETOXGR (toxicity grade, often 1–5) - AESTDTC (start date) - AEENDTC (end date) - TRTSDT (study treatment start date, for relatedness determination)

3. ADLB: Laboratory Analysis Dataset - One row per lab test per visit per subject - Built from SDTM LB domain - Added variables: - AVAL (analysis value, e.g., 142 for serum potassium) - BASE (baseline lab value) - CHG (change from baseline: AVAL - BASE) - PCHG (percent change: (CHG / BASE) × 100) - ANRIND (abnormality indicator: “L” for low, “H” for high, “N” for normal) - ABLFL (baseline flag, “Y” for baseline visit)

4. ADTTE: Time-to-Event Analysis Dataset - One row per subject - Built from SDTM domains (AE, LB, DM) and external events - Variables: - AVAL (time from baseline to event in days) - CNSR (censoring flag: 0 = event occurred, 1 = censored/did not occur by study end) - ADEVENT (event description) - EVENTDTC (event date)

7.4.3 ADaM Variable Conventions

Key ADaM variables follow strict patterns:

  • AVAL: The analysis value (primary outcome value at visit)
  • BASE: Baseline value of the same variable
  • CHG: Change from baseline = AVAL - BASE
  • PCHG: Percent change = (CHG / BASE) × 100
  • DTYPE: Data type flag
    • “C” = Calculated derived variable
    • “D” = Dummy record (added for consistency)
  • PARAM / PARAMCD: Parameter and parameter code
    • Example: PARAMCD=“SYSBP”, PARAM=“Systolic Blood Pressure”
  • ABLFL: Analysis baseline flag (“Y” = baseline, “N” or missing = not baseline)
  • ANRIND: Abnormality indicator for lab values (“L”, “H”, “N”)

7.5.3 Building ADSL: Subject-Level Analysis Dataset

# Step 1: Start with DM domain
adsl <- dm %>%
  select(STUDYID, USUBJID, AGE, SEX, RACE, ETHNIC, ARMCD, ARM) %>%
  rename(TRT01A = ARM,
         TRT01AN = ARMCD) %>%
  mutate(
    # Create subject-level identifiers
    SAFFL = "Y",  # All subjects are in safety population
    EFFFL = ifelse(AGE >= 18, "Y", "N"),  # Efficacy pop: age >= 18

    # Add baseline characteristics
    AGEGR1 = case_when(
      AGE < 40 ~ "<40",
      AGE >= 40 & AGE < 65 ~ "40-64",
      AGE >= 65 ~ ">=65"
    ),

    SEXN = ifelse(SEX == "M", 1, 2),

    # Add treatment dates (from EX domain)
    TRTSTDT = min(as.Date(ex$EXSTDTC), na.rm = TRUE),
    TRTENDT = max(as.Date(ex$EXENDTC), na.rm = TRUE)
  ) %>%
  arrange(USUBJID)

7.5.5 Building ADLB: Lab Analysis Dataset

# Step 3: Build ADLB from VS domain (using vital signs as example labs)
# Correct SDTM VS columns: VSSTRESU (units), VSDTC (date), VSSTRESN (numeric result)
adlb <- vs %>%
  select(STUDYID, USUBJID, VSTESTCD, VSTEST, VSSTRESN, VSSTRESU,
         VISITNUM, VISIT, VSDTC) %>%
  rename(PARAMCD = VSTESTCD,
         PARAM   = VSTEST,
         AVAL    = VSSTRESN,
         UNIT    = VSSTRESU) %>%
  mutate(
    # Parse date from ISO 8601 character
    VISITDT = as.Date(VSDTC),
    AVAL    = as.numeric(AVAL),

    # Add analysis flags
    ABLFL = ifelse(VISITNUM == 0, "Y", "N"),  # Baseline flag
    DTYPE = NA_character_                       # Not a calculated variable (raw SDTM)
  )

# Merge with baseline values
adlb <- adlb %>%
  left_join(
    adlb %>%
      filter(ABLFL == "Y") %>%
      select(USUBJID, PARAMCD, AVAL) %>%
      rename(BASE = AVAL),
    by = c("USUBJID", "PARAMCD")
  ) %>%
  mutate(
    # Derive change from baseline
    CHG = ifelse(is.na(BASE), NA_real_, AVAL - BASE),
    PCHG = ifelse(is.na(BASE) | BASE == 0, NA_real_,
                  (CHG / BASE) * 100)
  )

7.6 5.5 Metadata and Traceability

7.6.1 Why Metadata Matters

In a regulatory submission, the FDA asks: “Where did this number come from?” Metadata—data about data—provides the answer.

Essential metadata for each ADaM variable: - Variable name and label: AVAL = “Analysis Value” - Definition: How was it calculated? - Source: Which SDTM variables contributed? - Derivation: Formula (e.g., CHG = AVAL - BASE) - Validation rules: What values are valid?

# Example: Create a metadata data frame for ADLB
adlb_metadata <- tibble::tribble(
  ~Variable, ~Label, ~Definition, ~Source_Domain, ~Derivation,
  "USUBJID", "Unique Subject ID", "Subject identifier", "SDTM AE", "From AE domain",
  "PARAMCD", "Parameter Code", "Lab test code", "SDTM VS", "VSTESTCD from VS",
  "AVAL", "Analysis Value", "Result value at visit", "SDTM VS", "VSSTRESN from VS",
  "BASE", "Baseline Value", "Value at baseline visit", "SDTM VS", "AVAL where VISITNUM=0",
  "CHG", "Change from Baseline", "Difference from baseline", "SDTM VS", "AVAL - BASE",
  "PCHG", "Percent Change", "Percent difference from baseline", "SDTM VS", "(CHG/BASE)*100"
)

When Moderna submitted data for the COVID-19 vaccine trial to the FDA in late 2020, every table, figure, and listing (TFL) submitted was built from standardized data formatted according to CDISC standards. 1. **Ensure regulatory acceptance**: FDA, EMA, PMDA, and Health Canada accept CDISC-formatted data
2. **Enable data sharing**: Multiple sites, CROs, and statisticians work with consistent definitions
3. **Facilitate audits**: Regulators and auditors can trace any analysis result back to source data
4. **Reduce transcription errors**: Automated validation catches inconsistencies
5. **Enable re-use**: Future analyses (meta-analyses, real-world studies) can leverage existing data

## 5.2 SDTM: Study Data Tabulation Model

### Overview

The **SDTM** is the FDA-preferred format for submitting raw and derived clinical data. Think of SDTM as the "source of truth"—it documents exactly what happened in the trial, with minimal interpretation.

**Key principles:**
- **One observation per row**: Each row represents one test, one adverse event, one medication administration, etc.
- **Domains**: Data is organized by type (demographics, adverse events, labs, vital signs)
- **Standardized variable naming**: Variables follow consistent patterns
- **No derived calculations**: Values are directly from the case report form (CRF) or automatically captured systems
- **Traceability**: Each value can be traced back to its source

### SDTM Domains

Major domains include:

| Domain | Full Name | Content | Rows |
|--------|-----------|---------|------|
| **DM** | Demographics | Age, sex, ethnicity, enrollment date | One per subject |
| **AE** | Adverse Events | All reported adverse events | One per event |
| **LB** | Laboratory | All lab test results | One per test per visit |
| **VS** | Vital Signs | Blood pressure, heart rate, temperature | One per type per visit |
| **EX** | Exposure | Drug doses, administration times | One per dose per day |
| **CM** | Concomitant Medications | Other drugs taken during trial | One per medication per date range |
| **MH** | Medical History | Pre-trial medical conditions | One per condition |
| **SV** | Subject Visits | Planned visits | One per visit |
| **PE** | Physical Examination | Exam findings | One per exam |

For our hypertension example, key domains are **DM** (baseline characteristics), **VS** (blood pressure readings), **LB** (kidney function, potassium), **AE** (side effects), **EX** (drug doses), and **CM** (background medications like other antihypertensives).

### SDTM Variable Naming Conventions

SDTM uses a systematic naming structure:

**Prefix–Suffix pattern:**
- **STUDYID**: Study identifier (e.g., "HT-2024-001")
- **USUBJID**: Unique subject identifier (e.g., "HT-2024-001-0042")
- **--STDTC** / **--DTC**: Start/occurrence date-time in ISO 8601 format (YYYY-MM-DD'T'HH:MM:SS)
  - Example: "2024-03-15" or "2024-03-15T09:30:00"
- **--DTCF**: Date imputation flag (shows if date was complete or imputed)
- **DOMAIN**: Two-letter domain code (DM, AE, LB, etc.)
- **--SEQ**: Sequence number (row ID within domain for a subject)
- **--TESTCD**: Short test code (e.g., "SYSBP" for systolic BP)
- **--TEST**: Long test name (e.g., "Systolic Blood Pressure")
- **--RESULT**: The actual value
- **--UNIT**: Unit of measurement (e.g., "mmHg", "g/dL")
- **--VISIT**: Visit name (e.g., "Week 0", "Week 4")
- **--VISITNUM**: Visit sequence (0 = baseline, 1 = week 2, etc.)

**Example**: In the VS domain for systolic blood pressure:
- **VSDTC** = "2024-03-15T09:30:00" (date/time of measurement — ISO 8601)
- **VSTEST** = "Systolic Blood Pressure" (full test name)
- **VSTESTCD** = "SYSBP" (short code)
- **VSSTRESN** = 148 (standardised numeric result)
- **VSSTRESU** = "mmHg" (standardised unit)

This structure ensures any analyst can understand the data without additional documentation.

::: {.callout-tip}
## CDISC Controlled Terminology
The CDISC Controlled Terminology (CT) defines allowed values for many variables. For example, SEXN (sex numeric) uses 1 = Male, 2 = Female. The CT is updated annually; always reference the correct version used in your trial protocol.
:::

## 5.3 ADaM: Analysis Data Model

### Overview and Relationship to SDTM

While SDTM captures raw data, analysis requires *derived* variables. This is where **ADaM** comes in.

**ADaM** (Analysis Data Model) is built *from* SDTM but adds:
- **Derived variables**: Calculated fields like change from baseline, treatment-emergent status
- **Subject-level data**: Demographics merged with outcomes
- **Analysis-ready format**: One row = one analysis unit (subject, subject-visit, subject-event)
- **Audit trail**: Variables show which SDTM records contributed to each ADaM row

**The SDTM → ADaM → TLF flow:**

```
Raw Data (CRF, Labs, EHR)
        ↓
SDTM (standardized raw data, one observation per event)
        ↓
ADaM (derived analysis data, one row = one analysis unit)
        ↓
TLF (Tables, Listings, Figures for submission)
```

### Key ADaM Datasets

**1. ADSL: Subject-Level Analysis Dataset**
- One row per subject
- Contains: demographics, baseline characteristics, treatment assignment, trial status
- Example variables:
  - USUBJID, STUDYID
  - TRT01A (actual treatment assigned)
  - TRT01PN (treatment sequence number)
  - TRTSTDT (treatment start date)
  - TRTENDT (treatment end date)
  - SAFFL (safety population flag, 1 if in analysis)
  - EFFFL (efficacy population flag)
  - DTHFL (death flag) / RFENDTC (end of reference period - used to derive discontinuation)
  - BASE (baseline value of primary outcome)
  - AGE, SEX, ETHNIC, RACE

**2. ADAE: Adverse Events Analysis Dataset**
- One row per adverse event per subject (or one row per event for detailed analysis)
- Built from SDTM AE domain
- Added variables:
  - AESEV (severity: mild, moderate, severe)
  - AEREL (relationship to study drug: unrelated, remote, possible, probable, definite)
  - AESER (serious adverse event flag)
  - AETOXGR (toxicity grade, often 1–5)
  - AESTDTC (start date)
  - AEENDTC (end date)
  - TRTSDT (study treatment start date, for relatedness determination)

**3. ADLB: Laboratory Analysis Dataset**
- One row per lab test per visit per subject
- Built from SDTM LB domain
- Added variables:
  - AVAL (analysis value, e.g., 142 for serum potassium)
  - BASE (baseline lab value)
  - CHG (change from baseline: AVAL - BASE)
  - PCHG (percent change: (CHG / BASE) × 100)
  - ANRIND (abnormality indicator: "L" for low, "H" for high, "N" for normal)
  - ABLFL (baseline flag, "Y" for baseline visit)

**4. ADTTE: Time-to-Event Analysis Dataset**
- One row per subject
- Built from SDTM domains (AE, LB, DM) and external events
- Variables:
  - AVAL (time from baseline to event in days)
  - CNSR (censoring flag: 0 = event occurred, 1 = censored/did not occur by study end)
  - ADEVENT (event description)
  - EVENTDTC (event date)

### ADaM Variable Conventions

Key ADaM variables follow strict patterns:

- **AVAL**: The analysis value (primary outcome value at visit)
- **BASE**: Baseline value of the same variable
- **CHG**: Change from baseline = AVAL - BASE
- **PCHG**: Percent change = (CHG / BASE) × 100
- **DTYPE**: Data type flag
  - "C" = Calculated derived variable
  - "D" = Dummy record (added for consistency)
- **PARAM / PARAMCD**: Parameter and parameter code
  - Example: PARAMCD="SYSBP", PARAM="Systolic Blood Pressure"
- **ABLFL**: Analysis baseline flag ("Y" = baseline, "N" or missing = not baseline)
- **ANRIND**: Abnormality indicator for lab values ("L", "H", "N")

## 5.5 Metadata and Traceability

### Why Metadata Matters

In a regulatory submission, the FDA asks: *"Where did this number come from?"* Metadata—data about data—provides the answer.

**Essential metadata for each ADaM variable:**
- **Variable name and label**: AVAL = "Analysis Value"
- **Definition**: How was it calculated?
- **Source**: Which SDTM variables contributed?
- **Derivation**: Formula (e.g., CHG = AVAL - BASE)
- **Validation rules**: What values are valid?

```{r}
# Example: Create a metadata data frame for ADLB
adlb_metadata <- tibble::tribble(
  ~Variable, ~Label, ~Definition, ~Source_Domain, ~Derivation,
  "USUBJID", "Unique Subject ID", "Subject identifier", "SDTM AE", "From AE domain",
  "PARAMCD", "Parameter Code", "Lab test code", "SDTM VS", "VSTESTCD from VS",
  "AVAL", "Analysis Value", "Result value at visit", "SDTM VS", "VSSTRESN from VS",
  "BASE", "Baseline Value", "Value at baseline visit", "SDTM VS", "AVAL where VISITNUM=0",
  "CHG", "Change from Baseline", "Difference from baseline", "SDTM VS", "AVAL - BASE",
  "PCHG", "Percent Change", "Percent difference from baseline", "SDTM VS", "(CHG/BASE)*100"
)

print(adlb_metadata)
```

## 5.8 Practical Example: A Complete SDTM-to-ADaM Workflow

Let's build a comprehensive example for a hypothetical hypertension trial:

```{r}
# ============================================
# COMPLETE WORKFLOW: Hypertension Trial
# ============================================

# 1. Load and inspect SDTM domains
head(dm[, c("USUBJID", "AGE", "SEX", "ARM")], 3)
head(ex[, c("USUBJID", "EXTRT", "EXSTDTC", "EXENDTC")], 3)
head(vs[, c("USUBJID", "VSTESTCD", "VSSTRESN", "VISITNUM")], 3)

# 2. Build ADSL (subject-level)
adsl_complete <- dm %>%
  select(STUDYID, USUBJID, AGE, SEX, RACE, ETHNIC, ARMCD, ARM) %>%
  mutate(
    # Treatment assignment
    TRT01A = ARM,
    TRT01AN = as.numeric(factor(ARMCD)),

    # Eligibility flags
    SAFFL = "Y",
    EFFFL = "Y",

    # Demographics
    AGEGR1 = case_when(
      AGE < 40 ~ "<40",
      AGE >= 40 & AGE < 65 ~ "40-64",
      AGE >= 65 ~ ">=65"
    ),
    SEXN = ifelse(SEX == "M", 1, 2),

    # Treatment dates
    TRTSTDT = as.Date("2024-01-15"),  # Common start
    TRTENDT = as.Date("2024-03-15")
  ) %>%
  arrange(USUBJID)

# 3. Build ADLB (vital signs as lab analogue)
adlb_complete <- vs %>%
  # Filter to key parameters
  filter(VSTESTCD %in% c("SYSBP", "DIABP", "PULSE")) %>%
  # SDTM VS: VSSTRESU = standardised unit; VSDTC = date of measurement
  select(STUDYID, USUBJID, VSTESTCD, VSTEST, VSSTRESN, VSSTRESU,
         VISITNUM, VISIT, VSDTC) %>%
  rename(PARAMCD = VSTESTCD,
         PARAM   = VSTEST,
         AVAL    = VSSTRESN) %>%
  mutate(
    AVAL    = as.numeric(AVAL),
    VISITDT = as.Date(VSDTC),
    ABLFL   = ifelse(VISITNUM == 0, "Y", "N")
  ) %>%
  # Add baseline values
  left_join(
    vs %>%
      filter(VSTESTCD %in% c("SYSBP", "DIABP", "PULSE"),
             VISITNUM == 0) %>%
      select(USUBJID, VSTESTCD, VSSTRESN) %>%
      mutate(VSSTRESN = as.numeric(VSSTRESN)) %>%
      rename(PARAMCD = VSTESTCD, BASE = VSSTRESN),
    by = c("USUBJID", "PARAMCD")
  ) %>%
  mutate(
    CHG = AVAL - BASE,
    PCHG = ifelse(BASE != 0, (CHG / BASE) * 100, NA_real_)
  ) %>%
  select(STUDYID, USUBJID, PARAMCD, PARAM, VISITNUM, VISIT, VISITDT,
         AVAL, BASE, CHG, PCHG, ABLFL)

# 4. Summary: Baseline characteristics
cat("=== BASELINE DEMOGRAPHICS ===\n")
adsl_complete %>%
  group_by(TRT01A) %>%
  summarise(
    N = n(),
    Mean_Age = mean(AGE),
    SD_Age = sd(AGE),
    Male_N = sum(SEX == "M"),
    .groups = 'drop'
  )

# 5. Summary: Baseline vitals by treatment
cat("\n=== BASELINE SYSTOLIC BP BY ARM ===\n")
adlb_complete %>%
  filter(PARAMCD == "SYSBP", ABLFL == "Y") %>%
  left_join(adsl_complete %>% select(USUBJID, TRT01A), by = "USUBJID") %>%
  group_by(TRT01A) %>%
  summarise(
    N = n(),
    Mean = mean(AVAL),
    SD = sd(AVAL),
    Min = min(AVAL),
    Max = max(AVAL),
    .groups = 'drop'
  )
```
